# กรอบเอกสารประกอบระบบตรวจสอบและรายงานพัสดุคอมพิวเตอร์ (System Documentation Framework)
**เทศบาลนครนครสวรรค์ • กองยุทธศาสตร์และงบประมาณ**

เอกสารฉบับนี้จัดทำขึ้นเพื่อกำหนดโครงสร้างเอกสารประกอบระบบ (System Documentation) สำหรับนำไปเสนอต่อผู้เชี่ยวชาญ คณะกรรมการตรวจรับพัสดุ และผู้ตรวจสอบภายใน (Internal Auditor) เพื่อรีวิวและประเมินความสอดคล้องตามระเบียบกระทรวงการคลังว่าด้วยการจัดซื้อจัดจ้างและการบริหารพัสดุภาครัฐ

---

## 1. เอกสารภาพรวมระบบและสถาปัตยกรรม (System Architecture Document)
*เอกสารสำหรับผู้เชี่ยวชาญด้าน IT และนักพัฒนา*

*   **วัตถุประสงค์:** อธิบายโครงสร้างทางเทคนิค การเลือกใช้เทคโนโลยี และแนวทางการออกแบบระบบเพื่อให้สอดคล้องกับการทำงานแบบไร้เซิร์ฟเวอร์ (Serverless) และปลอดภัยตามมาตรฐานภาครัฐ
*   **หัวข้อที่ต้องมี:**
    *   **Technology Stack:** รายละเอียด Frontend (React + Vite + Tailwind CSS), โครงสร้างการจัดเก็บข้อมูล (Local Database ด้วย JSON และ Client-side Storage ด้วย LocalStorage) และบริการโฮสติ้ง (Firebase Hosting)
    *   **Data Integrity & State Management:** กลไกการแชร์สถานะการตรวจรับออนไลน์ผ่าน URL Hash (Base64 Compression) ซึ่งช่วยให้กรรมการสามารถแชร์ข้อมูลล่าสุดให้กันได้โดยไม่ต้องเปิดเผยข้อมูลลงฐานข้อมูลส่วนกลางหรือระบบคลาวด์สาธารณะ (Data Privacy by Design)
    *   **Performance & Responsiveness:** รายละเอียดการออกแบบ Mobile-first Responsive UI สำหรับแท็บเล็ตและสมาร์ตโฟนในสนามจริง

## 2. คู่มือการใช้งานระบบสำหรับผู้ตรวจรับพัสดุ (User & Administration Manual)
*เอกสารสำหรับคณะกรรมการตรวจรับพัสดุและเจ้าหน้าที่พัสดุ*

*   **วัตถุประสงค์:** แนะนำขั้นตอนการใช้งานระบบอย่างเป็นลำดับขั้น (Step-by-Step) เพื่อเพิ่มประสิทธิภาพและความแม่นยำในการตรวจรับสินค้าจริงหน้างาน
*   **หัวข้อที่ต้องมี:**
    *   **ขั้นตอนการเข้าใช้งานแดชบอร์ด:** การอ่านสถิติเบื้องต้น งบประมาณพัสดุ เปอร์เซ็นต์การตรวจผ่าน และจำนวนรายการพัสดุที่ยังไม่มีรูปประกอบ
    *   **ขั้นตอนการตรวจรับรายชิ้นและการบันทึกโน้ต:** การเปลี่ยนสถานะเป็น "ผ่านการตรวจรับ" หรือ "อยู่ระหว่างตรวจสอบ" พร้อมระบบบันทึกโน้ต/ข้อสังเกตของกรรมการ
    *   **ขั้นตอนการจับคู่รูปถ่ายและอัปเดตหลักฐาน:** วิธีการจับคู่ภาพถ่ายพัสดุจริง (Photo Evidence) กับรายการในระบบเพื่อใช้เป็นหลักฐานประกอบรายงาน
    *   **ระบบแชร์สถานะ (State Sharing):** วิธีการกดปุ่ม "แชร์สถานะให้ผู้อื่น" เพื่อสร้างลิงก์สำหรับส่งให้คณะกรรมการท่านอื่นรีวิวและลงลายมือชื่อพัสดุออนไลน์
    *   **การส่งออกรายงาน (Exporting):** คู่มือการจัดทำรายงาน PDF/Print Layout และการดาวน์โหลดรายงาน Excel (SheetJS) เพื่อเสนอฝ่ายจัดซื้อ

## 3. เอกสารมาตรฐานโครงสร้างข้อมูลและการจับคู่ไฟล์พัสดุ (Data Schema & Image Mapping Specification)
*เอกสารสำหรับเจ้าหน้าที่ฐานข้อมูลและผู้ตรวจสอบระบบ*

*   **วัตถุประสงค์:** ชี้แจงโครงสร้างข้อมูล (Database Scheme) ของพัสดุ 49 รายการ เพื่อการรักษามาตรฐานของฐานข้อมูลและการพัฒนาต่อยอดเป็นระบบพัสดุกลางในอนาคต
*   **หัวข้อที่ต้องมี:**
    *   **Data Dictionary (procurementData.json):**
        *   `id`: รหัสลำดับพัสดุ (Primary Key)
        *   `name`: ชื่อรายการพัสดุตามสัญญา
        *   `spec`: คุณลักษณะเฉพาะของพัสดุ
        *   `qty`: จำนวนจัดซื้อ
        *   `unit`: หน่วยนับ
        *   `unit_price`: ราคาต่อหน่วย
        *   `total_price`: ราคารวมเงิน (คำนวณแบบ Auto-calculation)
        *   `division`: กลุ่มงานที่ขอเบิกใช้งาน
        *   `image`: ชื่อไฟล์รูปภาพของพัสดุ (สำหรับเชื่อมหลักฐาน)
    *   **Image File Indexing (มาตรฐานการตั้งชื่อไฟล์รูปภาพ):** ข้อกำหนดของชื่อไฟล์รูปภาพและการระบุตำแหน่งรูปภาพบนเซิร์ฟเวอร์ (เช่น `public/รูปภาพ/`) เพื่อให้การค้นหาพัสดุรวดเร็วและไม่โหลดทรัพยากรมากเกินไป

## 4. รายงานการตรวจสอบความถูกต้องพัสดุ (Audit & Compliance Verification Report)
*เอกสารสำหรับผู้ตรวจสอบภายใน (Internal Auditor) และผู้บริหารเทศบาล*

*   **วัตถุประสงค์:** ใช้เสนอเป็นเอกสารแนบท้ายเพื่อยืนยันว่าการตรวจรับครั้งนี้โปร่งใส มีหลักฐานชัดเจน และสามารถตรวจสอบย้อนกลับได้แบบรายชิ้น (Traceability)
*   **หัวข้อที่ต้องมี:**
    *   **สรุปอัตราการจับคู่รูปพัสดุ (Photo Evidence Coverage):** อัตราส่วนพัสดุที่มีรูปภาพถ่ายจริงประกอบยืนยันตัวตน (เช่น 48/49 รายการ พร้อมชี้แจงรายการที่ไม่มีรูปภาพ เช่น ปลั๊กโรลม้วน Miyawa sun เพื่อความโปร่งใส)
    *   **ตารางรายงานแนบท้ายคำสั่งตรวจรับพัสดุ:** รายงานแยกแยะตามกลุ่มงานผู้เบิก (สถิติ, บริหาร, ประชาสัมพันธ์, วิเคราะห์)
    *   **รายงานข้อสังเกตและโน้ตจากการตรวจรับ:** รายการพัสดุที่อยู่ระหว่างตรวจสอบ (Pending) หรือต้องการแก้ไขก่อนรับมอบ
    *   **คณะกรรมการผู้รับผิดชอบและวันลงนาม:** อ้างอิงคำสั่งเทศบาลนครนครสวรรค์ ที่ ๘๖๔/๒๕๖๙ ตรวจรับ ณ วันที่ 21 กรกฎาคม 2569 พร้อมช่องลงลายมือชื่อประธานและกรรมการตามกฎหมาย

## 5. คู่มือการติดตั้งและบำรุงรักษาระบบ (Deployment & Maintenance Guide)
*เอกสารสำหรับเจ้าหน้าที่ศูนย์สารสนเทศ (IT Department)*

*   **วัตถุประสงค์:** วิธีการจัดการระบบเบื้องหลัง (DevOps) การอัปเดตข้อมูล และการกู้คืนระบบเมื่อเกิดข้อผิดพลาด
*   **หัวข้อที่ต้องมี:**
    *   **Local Setup:** วิธีการ Clone โปรเจกต์, การรัน Local Dev Server ด้วย `npm run dev`
    *   **Database Updates:** วิธีการนำไฟล์ Excel ใบเสนอราคาใหม่มาผ่านการปรับปรุงข้อมูลและบันทึกเป็น `procurementData.json`
    *   **Firebase Hosting Deployment:** วิธีการจัดเตรียมระบบและคำสั่งรันอัปเดตด้วย `firebase deploy`
    *   **Troubleshooting & Data Resetting:** วิธีการรีเซ็ตระบบกลับเป็นค่าเริ่มต้นและการเคลียร์แคชข้อมูลบนบราวเซอร์
